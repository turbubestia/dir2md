# Issue 3 - Implementation Analysis for Loose Multi-Page Markdown Merging

## 1. Objective and Scope

This issue introduces a dedicated merge stage that consumes metadata JSON produced by `md-gen`, detects multi-page relationships for loose scanned pages, and produces a merge plan plus executable merge outputs.

Primary outcomes:
- Add a merge module in `src/md_mrg`.
- Produce one batch JSON in `mg-temp` containing an array of document objects and merge decisions.
- Preserve guaranteed PDF sequences (`is_verified_sequence=true`) without re-inference.
- Infer page ordering/grouping for loose image scans (`is_verified_sequence=false`) using summary + boundary continuity scoring.
- Execute merge actions from approved batch JSON:
  - merge fragment markdown into unified markdown per final document
  - merge source images into multi-page PDF per final document
  - generate merged-document summary

Out of scope for this issue:
- Backend/frontend UI wiring
- Model hosting/provisioning (llama.cpp remains external)
- Renaming policy beyond what the merge batch defines

## 2. Current State and Gap

Current `md-gen` behavior already provides most prerequisites:
- Per-source metadata JSON generation via `src/md_gen/metadata_writer.py`.
- `document_id` deterministic from source path.
- Fragment anchors (`first_line`, `last_line`) and per-page snippet summaries.
- PDF sources already grouped and marked with `is_verified_sequence=true`.
- LLM access exists in `src/md_gen/gateway.py` through `LlamaOcrGateway` and `LlamaSummaryGateway`.

Main gap:
- No module currently consumes all metadata JSON files to produce cross-file document assembly for loose pages.
- Current schema root is one document object; merge batch requires an array root (or compatibility wrapper).
- LLM connector is currently module-local to `md_gen`; `md_mrg` would otherwise duplicate connector logic.

## 3. Target Architecture (No Code)

### 3.1 Module Boundary

Add a new merge CLI module with two explicit workflows:
- `plan` workflow: read `md-temp` metadata JSON, infer merge groups/order, write batch plan to `mg-temp`.
- `apply` workflow: read batch plan JSON (possibly user-edited), produce merged markdown + PDF outputs.

Recommended internal layers:
- `batch_loader`: reads and validates metadata JSON set.
- `sequence_inference`: scores possible page continuations and resolves chains.
- `edge_scorer`: pluggable scoring interface used by `sequence_inference`.
- `llm_gateway_shared`: shared language-model connector reused by `md_gen` and `md_mrg`.
- `plan_writer`: writes deterministic merge plan JSON.
- `executor_markdown`: concatenates markdown fragments + merged front matter.
- `executor_pdf`: merges images into multipage PDF.
- `global_summary`: generates final document summary from ordered page snippets.

### 3.3 Shared Gateway Refactor

To avoid duplicate LLM connectors, move gateway responsibilities out of `md_gen` into a shared module and rename language-facing connector to `LlamaLanguageGateway`.

Refactor intent:
- Keep OCR transport and language transport behind one reusable abstraction surface.
- Reuse the current summary gateway behavior for merge scorer implementation B (`llm` scorer on port `8081`).
- Prevent divergence in retry/error/timeout handling between modules.

Recommended shape:
- Shared package/module location (example): `src/common/llama_gateway.py`.
- Core classes:
  - `LlamaVisionGateway` (image + text OCR style requests)
  - `LlamaLanguageGateway` (text-in/text-out and structured JSON scoring calls)
- Existing `md_gen` call sites should migrate to shared gateways.
- `md_mrg` scorer B should consume `LlamaLanguageGateway` only, not a separate connector.

Compatibility expectation:
- During migration, allow temporary import shims in `src/md_gen/gateway.py` if needed to minimize breakage.
- After migration stabilizes, keep one source of truth for transport logic.

### 3.2 CLI Shape

Keep an explicit, safe split:
- `md-mrg plan --md-temp-dir ... --mg-temp-dir ...`
- `md-mrg apply --merge-batch-file ... --output-dir ...`

Add scorer selection to planning:
- `md-mrg plan --edge-scorer heuristic`
- `md-mrg plan --edge-scorer llm`
- Exactly one scorer is active per run; scorer choice is made by command argument.

Why this split is important:
- Enables human-in-the-loop editing in UI before irreversible output write.
- Keeps inference and execution independently testable.

## 5. Sequence Inference Strategy for Loose Pages

Your prompt proposal is valid and can be operationalized with deterministic boundaries.

### 5.1 Candidate Generation (Prevent O(N^2) Explosion)

For `N` loose pages, unrestricted pairwise scoring is expensive. Use a two-stage strategy:
1. Fast pre-filter candidates with cheap lexical heuristics from anchors/snippets.
2. Run LLM bridge scoring only on shortlisted neighbors (rolling window `k`).

Design choice: scoring is swappable.
- Define one scoring interface and two implementations.
- Both implementations consume the same candidate edge list and return a normalized score contract.
- This allows controlled A/B measurement of accuracy and runtime.

Scoring interface contract (planning layer):
- Input:
  - ordered loose-page nodes
  - candidate edges generated from rolling window `k`
  - scorer config (`timeout`, `retries`, endpoint/model when applicable)
- Output per edge:
  - `from_fragment_id`, `to_fragment_id`
  - `score_0_10`
  - `reason`
  - `scorer_type` (`heuristic` or `llm`)
  - `latency_ms`
  - optional `feature_breakdown` (heuristic) or `raw_model_response` (llm)

Fixed decisions for this stage:
- Initial loose-page order seed: filename lexicographic (case-insensitive).
- Rolling window size: `k=5`.

Implementation detail for point 1 (pre-filter and shortlist generation):
1. Build a normalized page list:
  - Load only fragments from documents where `is_verified_sequence=false`.
  - Flatten to page-level candidate nodes with fields:
    - `fragment_id` (stable id from source document id + sequence + markdown filename)
    - `first_line`, `last_line`, `snippet`
    - `image_file`, `markdown_file`, `source_name`
  - Sort nodes by `image_file` filename lexicographic.
2. Generate directional local neighbors:
  - Because scans can be out of order, use a bidirectional local window.
  - For each page at position `i`, create candidate edges `A -> B` for any `j` where `1 <= |j-i| <= 5` and `j != i`.
  - This keeps comparisons local but allows recovery when one page was scanned earlier/later than expected.
  - This limits raw candidates to at most `N * 10`.
3. Compute heuristic features per edge `A -> B`:
  - Library choices:
    - `rapidfuzz` for robust lexical similarity (`token_set_ratio`, `partial_ratio`).
    - `re` and `unicodedata` from stdlib for normalization and entity extraction.
    - Optional `python-dateutil` for date normalization when date formats vary heavily.
  - Text normalization (applied to anchors and snippets before scoring):
    - unicode normalize (NFKC), lowercase, strip punctuation noise, collapse whitespace.
    - remove stopwords from a small fixed bilingual set (EN/ES) to reduce weak matches.
    - keep alphanumeric tokens and domain symbols relevant for bills/invoices (`-`, `/`, `#`).
  - `boundary_overlap` (0..10):
    - Primary signal for continuation from `A.last_line` to `B.first_line`.
    - Compute:
      - `s1 = rapidfuzz.fuzz.partial_ratio(A.last_line, B.first_line) / 100`
      - `s2 = jaccard(last_5_tokens(A.last_line), first_5_tokens(B.first_line))`
      - `boundary_overlap = 10 * (0.7*s1 + 0.3*s2)`
  - `summary_overlap` (0..10):
    - Semantic lexical proximity using snippet text.
    - Compute:
      - `s1 = rapidfuzz.fuzz.token_set_ratio(A.snippet, B.snippet) / 100`
      - `s2 = cosine_tfidf_char_3gram(A.snippet, B.snippet)` using a lightweight local vectorizer (or fallback to token Jaccard if TF-IDF is disabled).
      - `summary_overlap = 10 * (0.6*s1 + 0.4*s2)`
  - `numeric_entity_match` (0..10):
    - Extract entities via regex from both pages:
      - dates (`dd/mm/yyyy`, `yyyy-mm-dd`, month-name variants)
      - amounts/currency (`$123.45`, `123,45`, etc.)
      - ids (`invoice`, `account`, `service`, `ref`, long digit groups)
    - Normalize extracted values (date canonical form, amount decimal canonical form).
    - Compute weighted entity overlap:
      - `date_match` (weight 0.4), `id_match` (weight 0.4), `amount_match` (weight 0.2)
      - `numeric_entity_match = 10 * weighted_jaccard`
  - `layout_hint` (0..1 additive bonus):
    - Add `+0.5` when filename stems suggest same scan burst (common prefix + nearby serial number).
    - Add `+0.5` when image dimensions/aspect ratio are very close (same scanner profile).
    - Keep max `layout_hint = 1.0` so it never dominates textual evidence.
4. Combine heuristic features into a pre-score:
  - Example weighted formula:
    - `pre_score = 0.45*boundary_overlap + 0.35*summary_overlap + 0.20*numeric_entity_match + layout_hint`
  - Normalize to `0..10` for easy ranking.
  - Design intent:
    - boundary continuity remains strongest signal.
    - summary and entity consistency reduce false positives on repetitive templates.
    - layout is only a tie-break nudge.
5. Prune before LLM calls:
  - Keep only edges with `pre_score >= 2.5`.
  - If none pass for a page, keep top 1 edge as exploratory fallback to reduce false negatives.
  - Cap evaluated edges per page to max 3 after pruning.
6. Produce final LLM shortlist:
  - Sort candidate edges for each source page by `pre_score` descending, then lexical tie-break by target `fragment_id`.
  - Emit shortlist for bridge scoring stage.

### 5.1.7 Swappable Scorer Implementations

Implementation A: `heuristic` scorer (default for speed)
- Uses local text/entity features and weighted formula from section 5.1.
- No network calls.
- Expected behavior:
  - very fast and deterministic
  - may have lower semantic sensitivity on difficult continuations

Implementation B: `llm` scorer (local model on port 8081)
- Uses the rolling-window candidate edges and calls the local LLM endpoint for each edge.
- Endpoint/model source:
  - base endpoint from config (`http://localhost:8081/v1/chat/completions` by default)
  - model configured in merge module settings
- Prompt shape:
  - system prompt and score schema follow the request in [design/issues/3-merge-muti-page-ocr.plan.request.md](design/issues/3-merge-muti-page-ocr.plan.request.md)
  - input fields per edge: Page A End, Page A Summary, Page B Start, Page B Summary
- Output parser:
  - strict JSON parse of `{"reason": "...", "bridge_score": int}`
  - invalid responses mapped to scorer error and treated as `reject` unless retry succeeds
- Expected behavior:
  - higher semantic robustness
  - slower due to inference latency

Selection behavior:
- `heuristic`: use heuristic scores as `bridge_score` input to chain resolver.
- `llm`: use LLM `bridge_score` as input to chain resolver.

Gateway integration detail for implementation B:
- `llm` scorer will call `LlamaLanguageGateway` with the bridge prompt and strict JSON response parsing.
- The same timeout/retry/error mapping contract used by summary extraction should be reused.

Why this is deterministic and bounded:
- Sorting keys and tie-breakers are fixed.
- Candidate construction is window-limited (`k=5`).
- Pruning and per-page cap are fixed rules.
- Total LLM comparisons remain finite and predictable.

### 5.2 LLM Bridge Scoring

Use your scoring JSON (`reason`, `bridge_score`) with these decisions:
- `bridge_score >= 7`: auto-merge eligible.
- `bridge_score == 5 or 6`: merge and flag for review-required candidate (add the review property to the json schema).
- `bridge_score <= 4`: not mergeable (no continuity).

Store per evaluated edge:
- `from_fragment_id`, `to_fragment_id`
- `bridge_score`
- `reason`
- optional `heuristic_score`
- `decision_status` (`auto_merge`, `review_required`, `reject`)
- `scorer_type`
- `latency_ms`

### 5.3 Deterministic Chain Resolution

Treat pages as a directed graph where edges are candidate continuations.

Constraints:
- each page has at most one outgoing edge (next page)
- each page has at most one incoming edge (previous page)
- no cycles

Resolution algorithm (deterministic greedy):
1. Sort eligible edges by:
   - highest `bridge_score`
   - highest heuristic score
   - stable lexical tiebreak (`from_id`, `to_id` ascending)
2. Accept edge if it does not violate in/out degree or create a cycle.
3. Continue until edge list exhausted.

This yields disjoint chains representing inferred multi-page documents.

### 5.4 Exit Criteria (No Endless Comparison)

Deterministic stop conditions:
- Candidate set is finite and precomputed before scoring.
- Each candidate is scored at most once per run.
- Chain resolution is a single pass over sorted edges.

Upper-bound comparisons:
- with rolling window `k`: at most `N * k` LLM calls
- with full pairing: at most `N * (N - 1)`

So this is finite and not open-ended; conceptually graph assembly, not infinite sorting.

## 6. Handling Verified Sequences

For documents where `is_verified_sequence=true`:
- skip inference entirely
- carry fragments directly into final plan in sequence order
- mark `merge_confidence=1.0` or equivalent strong status

This keeps PDF-derived documents deterministic and cheap.

## 7. Apply Workflow Design

### 7.1 Markdown Merge

For each planned document:
- read fragment markdown files in final sequence order
- concatenate with explicit page boundary separator `---`
- prepend merged front matter with:
  - generation timestamp
  - source document ids/names
  - merge confidence
  - fragment count/page count
  - global summary
  - review status (when any edge in chain required review)

### 7.2 PDF Merge from Images

For each planned document:
- read image fragment assets in same sequence
- produce one multipage PDF in output folder

Library guidance:
- Prefer `img2pdf` for faithful image-to-PDF assembly.
- `Pillow` can work but may alter compression/metadata behavior depending on format.
- For pages with US-letter aspect ratio use US-letter size.
- For pages with US-legal aspect ratio use US-legal size.
- For any other aspect ratio, center-fit on US-letter canvas.

### 7.3 Global Summary for Merged Document

Generate final summary from ordered fragment snippets only.

Decision:
- Do not use full merged markdown as summary input to avoid long-document context inflation.

Store summary in batch JSON and merged markdown front matter.

### 7.4 Output Filename Strategy

Output filename should be proposed by LLM using merged summary and key page hints.

Decision:
- Filename format target: `date-subject-title`.
- `md_gen` summary prompt should be updated to prioritize extraction of identifying fields such as dates, service/rendered service context, medical/utility topic markers, and other disambiguating document cues.
- Apply sanitize rules for filesystem safety and deterministic normalization.

## 8. Failure Modes and Recovery

Failure handling policy should be explicit per document:
- Inference failure for one candidate does not abort full run.
- Unmergeable pages remain standalone documents in output plan.
- Apply stage should continue per document and emit structured failure records.

Idempotency expectations:
- planning reruns should produce same output for same inputs and model responses
- apply reruns should respect overwrite policy and skip existing artifacts when overwrite is false

### 9.1 Accuracy and Speed Comparison Plan

To compare methods later, capture per-run metrics for each scorer invocation:
- total evaluated edges
- accepted edges count
- median/p95 latency per edge
- total scoring wall-clock time
- optional labeled-set precision/recall/F1 for a run

This supports future manual comparison between heuristic and LLM runs without adding a dual-mode execution path.

## 10. Risks and Mitigations

Risk: bridge score threshold `>=7` may still over-merge weakly related pages in repetitive document sets.
Mitigation: keep explicit review flags, preserve editable batch plans, and include reasons per selected edge.

Risk: false negatives split true documents.
Mitigation: keep unmatched pages editable in batch plan for UI correction.

Risk: cost/time increases with many loose pages.
Mitigation: enforce rolling window `k` and heuristic pre-filter.

Risk: schema break for existing consumers.
Mitigation: evolve `src/md_gen/metadata-schema.json` in backward-compatible form and add schema-version field if needed.

## 12. Resolved Design Decisions

The following choices are now fixed for implementation:

1. Module naming
- Use package folder `src/md_mrg`.

2. Batch schema strategy
- Update `src/md_gen/metadata-schema.json` to support merge-batch and document structures.

3. Rolling window size
- Use default `k=5`.

4. Threshold policy
- `>=7`: auto-merge.
- `=5 or 6`: merge with review flag.
- `<=4`: no merge.

5. Ordering seed for loose pages
- Use filename lexicographic order.

6. Global summary source
- Use page snippets only.

7. Output naming
- LLM-proposed filename from merged summary and key identifying cues.
- Filename pattern target: `date-subject-title`.

8. Markdown boundary marker
- Use `---` as page separator.

9. Low-confidence handling
- Score 5 and 6 edges are included with explicit review flag in plan and output metadata.

10. Existing output behavior
- Keep `--overwrite` semantics: skip existing outputs unless overwrite is enabled.

11. Scorer module architecture
- Use a swappable scoring interface with two implementations:
  - `heuristic` (feature-based local scorer)
  - `llm` (local model scorer on port 8081)

12. Gateway reuse architecture
- Move connector logic outside `md_gen` and use shared `LlamaLanguageGateway` for merge scorer implementation B.
- Do not introduce a second, independent LLM connector for `md_mrg`.
