## Features and Requirements

- The code must be modularized:
    - One module as a cli tool to scan the directory and produce the markdown files (`md-gen`)
    - one module as a cli tool to parse the markdown and consolidate documents: merge images to pdf, merge individual markdown pages, and file renaming (`md-norm`)
    - one module for the web backend (`backend`)
    - one module for the web frontend (`frontend`)

    This would results in a folder structure like:
```
/src/md-gen
/src/md-norm
/src/backend
/src/frontend
```

- The `md-gen` cli should accept a folder path that contains the pdf and images. It will create a `im-temp` folder where the PDF will be converted to images and the images will be scalled, both images must meet the maximum image size accepted by LightOnOCR (described below). It should also accept single or list of pdf or images. The output markdown files will be write to the folder `md-temp`. The markdown generated by the OCR will be augmented with a metadata header with at least information of the source document. For this task we will use the LightOnOCR model.

- The `md-norm` cli should accept a folder with the generated markdown (i.e. the `md-temp` generated by `md-gen`). It will read the md files and generate a summary or compress information that could allow to find out wich pages correspond to the same document. For multi-page pdf, `md-gen` could add a metadata to explicitly tell this information. For separated scanned images this must be discovered. For this task we will use the Gemma-4-E4B model. We should be able to specify a file name format for the system to propose.

- The frontend should give the user the option to select the source path with the document and the output path were the result will be place. The backend should move/clean the files after process them.

- The frontend will have to the left the list of files in the selected path next to a preview of the selected image or pdf (if possible).

- In the center should be a tree showing the result of the `md-norm` analysis to know the filename and which pages would be merged before commiting to the process.

- after the documents are process, next we will have a preview of the markdown files and to the right the list of generated markdowns.

In sumary the minimal frontend layout would be

| Source dir files (pdf or images only) | selected file preview | process preview (filename and merge as a tree) | preview of selected markdown | list of markdown file |

The three list should be synchronized allowing to compare source to result.

On top we can have a line edit and button to select the folder, and buttons to move between OCR, Norm, and Process.

# LightOnOCR-2 Token & Context Specifications

This document outlines how images are processed into tokens by LightOnOCR-2 and provides the metrics needed to calculate context window usage for your PDF/image parsing pipeline.

## 1. Core Architecture Metrics

LightOnOCR-2 uses a Vision Transformer (ViT) backbone coupled with a language decoder to parse visual layouts into Markdown text.

* **Base Patch Size:** 16 x 16 pixels.
* **Projector Mechanism:** Factor-2 Spatial Merging (groups patches into 2 x 2 blocks).
* **Max Image Dimension:** 1,540 pixels on the longest edge (images exceeding this are dynamically downscaled).
* **Total Context Window:** 16,384 tokens (16k) absolute limit for the language model decoder.

## 2. Token Reduction Pipeline

1. **Patch Generation:** An image is initially sliced into raw 16x16 pixel patches.
2. **Spatial Merging:** The multimodal projector compresses these patches by a factor of 4x. 
3. **Sequence Length:** The resulting compressed patches are mapped directly as sequence tokens into the 16k context window.

## 3. Context Token Formula

Because of the 4x reduction from the spatial merging step, the relationship between an image's pixel dimensions and its context token consumption is linear.

### Mathematical Formula
$$\text{Image Tokens} = \frac{\text{Width} \times \text{Height}}{1024}$$

### Code Implementation Example (Python)
```python
def calculate_vision_tokens(width: int, height: int) -> int:
    """Calculates the context window token cost for a given image size."""
    # Max longest edge check based on LightOnOCR-2 limits
    longest_edge = max(width, height)
    if longest_edge > 1540:
        scale = 1540 / longest_edge
        width = int(width * scale)
        height = int(height * scale)
        
    return int((width * height) / 1024)
