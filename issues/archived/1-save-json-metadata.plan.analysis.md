# Issue 1 - Save JSON metadata along with MD files (Implementation Analysis)

## 1. Objective and Scope

This issue replaces per-markdown front matter metadata with explicit JSON metadata artifacts that describe document structure and page-level fragments.

Primary outcomes:
- Stop writing provenance front matter into markdown outputs.
- Write one JSON metadata file per logical source document.
- For multi-page PDFs, preserve known page sequence in one JSON file.
- For loose single images, write one JSON file per image with sequence marked as not verified.
- Add page-level short summaries generated by a second LLM endpoint.
- Default endpoints should work without needing frequent CLI overrides.

Out of scope for this issue:
- Document merging behavior in md-norm.
- Any backend/frontend integration.
- Any model hosting changes (llama.cpp remains external and user-managed).

## 2. Current State Assessment

Current md-gen behavior:
- OCR endpoint/model are configured as a single model setting and currently default to 127.0.0.1:8080 with OpenAI-compatible chat completions path.
- OCR markdown files are persisted with deterministic hashed filenames.
- Markdown persistence currently prepends YAML-like front matter metadata.
- PDF page linkage is inferred only by each markdown file metadata fields, not by grouped document JSON.
- Existing schema file already models a document/fragments shape compatible with your target design, but it is not currently used by runtime persistence.

Main gap:
- Runtime produces markdown-only records and does not emit JSON document metadata files.

## 3. Desired Target Behavior

### 3.1 Artifact model

For each source logical document:
- Persist markdown files (one per fragment/page) in md-temp.
- Persist one JSON file in md-temp that references all fragment files for that source document.

For PDF input:
- One JSON per source PDF.
- total_pages equals PDF page count.
- is_verified_sequence equals true.
- fragments ordered by source page number.

For single image input:
- One JSON per source image.
- total_pages equals 1.
- is_verified_sequence equals false.
- one fragment entry.

### 3.2 Summary extraction

After OCR text is obtained for each fragment:
- Send OCR markdown text to Qwen summary endpoint.
- Parse strict structured output and extract concise summary text.
- Store summary in fragment content_fingerprint.snippet.

### 3.3 Anchors

For each fragment, populate anchors with at least:
- first_line
- last_line

Optional headers/footers can remain empty unless robustly detected in this phase.

## 5. Data Contract Mapping

Your proposed JSON schema maps well to runtime generation with minimal additions.

Mapping recommendation:
- document_id: stable UUID5 or deterministic hash from source path + run salt policy (decision needed).
- source_name: original input filename.
- total_pages: number of fragments in this JSON.
- is_verified_sequence:
  - true for PDF-derived fragments
  - false for standalone image-derived fragments
- fragments[].sequence_number:
  - PDF page number
  - 1 for standalone image
- fragments[].image_file: basename of processed image file used for OCR.
- fragments[].markdown_file: basename of markdown file.
- fragments[].anchors.first_line: extracted first meaningful line from OCR markdown.
- fragments[].anchors.last_line: extracted last meaningful line from OCR markdown.
- fragments[].content_fingerprint.snippet: summary output from Qwen.
- fragments[].content_fingerprint.detected_entities: initially optional/empty unless explicitly requested now.

## 6. Anchor Extraction Strategy

A deterministic normalization policy is needed to avoid unstable anchors.

Recommended baseline policy:
- Strip empty lines.
- Ignore markdown fence delimiters.
- Keep heading lines as valid content unless you want heading-skipping.
- first_line: first non-empty normalized line.
- last_line: last non-empty normalized line.
- If line length exceeds threshold, truncate by words (policy decision required).

Why this matters:
- md-norm relationship inference depends on consistent anchors.

## 7. Summary LLM Contract and Parser Safety

Your prompt enforces strict parser-safe output. The request currently says response must start with { and end with }.

Implementation analysis recommendation:
- Require summary endpoint to return strict JSON object with exactly one required string field, for example:
  - {"summary":"..."} **User: I removed this requirement, the return should be just pure text, no formating nor json.**
- Keep temperature at 0 for determinism.
- Enforce local max length post-parse as a safety belt. **User: don't trim the summary response.**

Operational note:
- If Qwen returns additional keys, keep in the summary only [a0zA-Z0-9 -.] to make it stable, basically only standard text characters.
