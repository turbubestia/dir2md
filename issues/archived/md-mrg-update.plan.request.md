# MD-MRG (Markdown Merge) Update From Scratch

In this task we will write the `src/md_mrg` module from scratch (the content was already deleted). The OCR procress from PDF or images is split in two big steps: (PDF|image) to Markdown, and then loose pages from images to merged Markdown. The first step is acomplish by the project `src/md_gen` and should be left untouched (it out of the scope of this task). This task deal with the second step and must be implemented in the folder `src/md_mrg`.

## Merge Pipeline Description

The merge of the loose pages originated from loose scanned images is in turn split in two steps: planning and merging. In a high level view the planner will read the `batch.json` file generated by `md_gen` and will compare by pair of files (with a scheme described below) and will put together those were found to belong to the same document. This will generate a new `batch_mrg.json` file. Later in the project we will present the content of this `batch_mrg.json` to the user in a UI to give him the change to review the result. When the ordering is approve, the step two os applying the mergin start. It will merge the corresponding markdown files together, and the original images will be converted to a single PDF. Original PDF will be moved to the output folder. The result will be each PDF (originals or merged images) will be accompanied by a markdown file with the OCR text of the corresponding file.

### CLI Arguments

- As arguments we can receive
    --source this correspond to the --output option of `md_gen`, where the batch.json and markdown document are

### Configuration

We will use the module `src/common/config.py` to read the configuration. The laguage model for the score of pages is the same use by the md_gen module in the `language_model` configuration section. There is a section `md_mrg` with specific configuration for this module.

### Planner stage

1. We read the batch.json located in the provided source folder and extract the document that are loose images, `file_type` in the batch as `image` type (so we leave PDF page OCR alone since thay are already merged automatically by the md_gen module). See `src\md_gen\page_processor.py` for details of each json document element.

2. Usually pages are scanned sequentially, so we only need to compare adjacent pair of pages. So we start to compare page i with i+1, if they belong together with |score| => 5 from the LLM score model we add them as an array and then we compare the i+1 with i+2 and so on until we find a page that does not belong to this group (|score| < 5).

If `d` is the type that holds the page information, and we called `d1`, `d2`, ..., `dn` pages 1,2 until n, here is a sequence of merging:

Lets assume we have 5 pages, {d1}, {d2}, {d3}, {d4}, {d5}

1. d1 <=> d2 == 6, then we group {d1, d2}, {d3}, {d4}, {d5}
2. d2 <=> d3 == 8, then we group {d1, d2, d3}, {d4}, {d5}
3. d3 <=> d4 == 2, then we skip {d1, d2, d3}, {d4}, {d5}
4. d4 <=> d5 == 7, then we group {d1, d2, d3}, {d4, d5}
5. done

So we would have two goups of documents: document 1 {d1, d2, d3}, and document 2 {d4, d5}

From the original `batch.json` we take the `pdf` types of document and append them to `batch_mrg.json`, with the found groups, i.e. {d1, d2, d3} and {d4, d5}, at the top.

{
    {d1, d2, d3},
    {d4, d5},
    d6,
    d7
}

where d6 and d7 would have been pdf documents.
This is the end of the plan stage.

### Apply stage

After the user has review the `batch_mrg.json` and made the necesary changes (by hand or in the future UI), we can start merging the document groups. We traverse the json and look for the groups, i.e. {d1, d2, d3}, and merge their markdown files into one and their original files into one PDF (then we delete only the loose markdown files). Theoriginal PDF are move into this folder.

### Original images

We left the responsability to the user to delete the images if he find the merging is correct.

## Propose structure

We will have a `cli.py` that is the entry point of the module. Here we will have either `--plan` or `--apply` flags to chose between both workflows (this flags are mandatory). The other mandatory argument is `--source`. The we will have a `planner.py` and `apply.py` modules for the implementation of each workflows.

## Note on the LLM Score system

In the `src/common/gateway` is the `LlamaLanguageGateway` that needs to be use to sent the score system prompt and both pages. Both pages needs to be plave in the user text with an structure like:

```
# Page A content
--- start of Page A content ---
{insert here page A text}
--- end of Page A content ---

---

# Page B content
--- start of Page B content ---
{insert here page B text}
--- end of Page B content ---

```

- **Technical Constraints / Edge Cases:**
