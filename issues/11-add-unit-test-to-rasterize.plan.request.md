# Refinement Iteration 1

## Context
User: Update, fix, and extend the unit test for the module `src/md_gen/rasterizer.py` located in `test/md_gen/test_rasterizer.py`. The test only covers the rasterizer module, so do not plan to test any other file. Nothing else. The test should create the source PDF files for testing. We look for full code coverage with the tests.

## Objective
Define complete, unambiguous requirements to revise `test/md_gen/test_rasterizer.py` so it fully validates behavior in `src/md_gen/rasterizer.py` with module-focused, self-contained unit tests.

## In Scope
- Changes to `test/md_gen/test_rasterizer.py`.
- Test logic that exercises behavior implemented in `src/md_gen/rasterizer.py`.
- Runtime creation of source PDF inputs during test execution.

## Out of Scope
- Tests for any module other than `src/md_gen/rasterizer.py`.
- Production code changes outside what is strictly required to enable or stabilize rasterizer test coverage.
- New integration or end-to-end tests spanning multiple modules.

## Requirements

### R1. Rasterizer-Only Coverage Target
- Description:
  - The revised test file must validate only behavior belonging to `src/md_gen/rasterizer.py`.
- Constraints:
  - Do not add planned assertions for `cli`, `config`, `discovery`, `resizer`, or any other module.
  - Any helper usage from other modules must be incidental, not a test target.
- Edge cases:
  - If rasterizer behavior includes interactions (filesystem, subprocess, libraries), tests should isolate these interactions while still asserting rasterizer-owned decisions and outputs.

### R2. Full Coverage for rasterizer.py
- Description:
  - The test suite in `test/md_gen/test_rasterizer.py` must exercise all executable code paths in `src/md_gen/rasterizer.py`.
- Constraints:
  - Target 100% statement coverage for `src/md_gen/rasterizer.py`.
  - Cover both success and failure branches, including error handling and boundary conditions.
- Edge cases:
  - Invalid or missing input files.
  - Empty or malformed outputs from rasterization dependencies.
  - Exceptional conditions raised by external calls (for example, conversion failures, filesystem write errors, unsupported input).

### R3. Source PDFs Must Be Generated by Tests
- Description:
  - Tests must create their own source PDF files at runtime instead of relying on pre-existing static fixtures.
- Constraints:
  - PDF creation must happen inside test setup (temporary directories/files).
  - Tests must be deterministic and independent of repository-local binary fixtures.
  - Generated artifacts must be cleaned up automatically by test lifecycle.
- Edge cases:
  - Single-page and multi-page generated PDFs where relevant to rasterizer logic.
  - Minimal valid PDF content sufficient to trigger rasterization paths.

### R4. Deterministic and Isolated Test Behavior
- Description:
  - Tests must be stable across runs and machines.
- Constraints:
  - Avoid dependence on global state, wall-clock timing assumptions, or non-deterministic ordering.
  - Mock or stub external process/library behavior when required to keep results deterministic.
- Edge cases:
  - Different filesystem separators and temp-path behavior on Windows.
  - Existing output directories/files from previous test runs must not affect results.

### R5. Assertions Quality
- Description:
  - Assertions must validate observable rasterizer outcomes, not just that functions were called.
- Constraints:
  - Verify expected output files/paths, return values, and raised exceptions.
  - Validate side effects in temporary output locations where rasterizer writes artifacts.
- Edge cases:
  - Partial output generation before failure.
  - Repeated invocations with identical inputs.

## Acceptance Criteria
- `test/md_gen/test_rasterizer.py` is updated and remains focused exclusively on `src/md_gen/rasterizer.py` behavior.
- Tests generate needed PDF inputs during execution.
- Coverage report indicates full code coverage for `src/md_gen/rasterizer.py`.
- Tests pass reliably in local project environment on Windows.

## Open Design Choices
1. Should tests prefer constructing minimal valid PDFs directly in Python, or generating PDFs through an allowed lightweight test dependency? (Choose one approach for consistency.)
2. If rasterizer depends on external binaries/services, should tests mock those calls entirely, or include optional real-execution tests guarded by availability checks?
